# Phase 7 -- UI Design Contract

> Visual and interaction contract for the Streamlit Dashboard. Generated by gsd-ui-researcher, verified by gsd-ui-checker.
>
> This phase is a **data dashboard** -- designed for information display and exploration, not form entry or CRUD.

---

## Design System

| Property | Value |
|----------|-------|
| Tool | none (Streamlit native -- shadcn/Radix not applicable) |
| Preset | not applicable |
| Component library | Streamlit native (`st.columns`, `st.expander`, `st.metric`, `st.spinner`, `st.progress`, `st.error`, `st.warning`, `st.button`) |
| Icon library | none (Streamlit emoji shorthand via `st.metric(delta=None)` and Unicode block characters for score bars) |
| Font | Streamlit default -- "Source Sans Pro", sans-serif (configured via `.streamlit/config.toml` `font = "sans serif"`) |
| Theme config | `.streamlit/config.toml` -- custom dark theme |

### Streamlit Theme Contract

Configured via `.streamlit/config.toml` `[theme]` section:

| Property | Value | Source |
|----------|-------|--------|
| `base` | `"dark"` | User decision |
| `primaryColor` | `#3498db` (blue -- neutral accent distinct from grade colors, used for button/input focus states only) | Default (Streamlit dark theme standard) |
| `backgroundColor` | `#0e1117` (near-black, Streamlit dark default) | Default |
| `secondaryBackgroundColor` | `#1a1d24` (dark card surface, Streamlit dark default) | Default |
| `textColor` | `#e0e0e0` (near-white, Streamlit dark default) | Default |
| `font` | `"sans serif"` | Default |

Note: Streamlit supports exactly one `primaryColor`. Grade colors (green/blue/yellow/orange/red) are semantic and applied via inline HTML/CSS in the app, not via the theme config. The `primaryColor` is reserved for the Analyze button and text input focus ring.

## Typography

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Body | 16px | 400 (regular) | 1.5 |
| Label | 14px | 600 (semibold) | 1.4 |
| Heading | 24px | 600 (semibold) | 1.2 |
| Display | 48px | 700 (bold) | 1.1 |

Usage mapping:
- **Body (16px/400/1.5):** All paragraph text, recommendation messages, expander content, general UI copy
- **Label (14px/600/1.4):** Module names in score cards, "Priority" labels in recommendations, metric labels (e.g., "Overall Score", "Grade"), table column headers
- **Heading (24px/600/1.2):** Module section titles ("Robots.txt", "llms.txt", "Schema", "Content"), "Recommendations" section header
- **Display (48px/700/1.1):** Grade letter (A/B/C/D/F) in the score hero -- single character, large and bold

Note: Streamlit's native `st.metric()` applies its own styling to the metric value. The Display size is for a custom-rendered grade badge via `st.markdown()` with HTML/CSS, not for `st.metric()` values.

### Score Bar Colors

Score bars use Streamlit's `st.progress()` which renders as a filled bar. The color is Streamlit's default blue accent (from `primaryColor`). Individual module score bars do NOT use grade colors -- grade colors are reserved for the grade badge only. When shown in expanders, score bars use `st.progress(score_value)` with Streamlit's default progress bar styling.

Accent reserved for: Analyze button, URL input focus ring, loading spinner. NOT used for grade badges, score bars, or any other interactive element.

## Copywriting Contract

| Element | Copy |
|---------|------|
| Primary CTA | **Analyze** (button label, triggers `run_pipeline()`) |
| URL input label | **Enter a URL to analyze** (placeholder text in `st.text_input()`) |
| Empty state heading | No heading shown -- url input only |
| Empty state body | Results area is completely blank. Only the URL input with placeholder and the Analyze button are visible. No "Enter a URL" prompt, no "Results will appear here" message. |
| Loading state | Streamlit native spinner: `st.spinner("Analyzing...")` with the text "Checking robots.txt, llms.txt, schema, content, and generating your score..." |
| Error state banner | Red `st.error()` callout at top: "Analysis failed for some stages. See details below." |
| Error state inline | For pipeline errors: error text shown inline in the results area under a red "Pipeline Errors" heading, listing each error message. URL input remains visible for retry. |
| Error state -- no URL | If user clicks Analyze with empty URL: "Please enter a URL." via `st.warning()` |
| Error state -- crawl failure | If crawl stage fails: "Could not reach {url}. {error_message}" displayed in the results area. Schema and Content modules show "Not available -- crawl failed." |
| Recommendations heading | **Recommendations** (h2, displayed only when recommendations list is non-empty) |
| Recommendations empty | Nothing displayed -- the Recommendations section is not rendered at all when the list is empty |
| Module expander labels | **Robots.txt** (score: X.XX) -- for all four modules: "Module Display Name (score: X.XX)" |
| Destructive actions | None in this phase. No delete, reset, or irreversible operations. |

### Module Detail Content (per expander)

When a module expander is opened:

- **Robots.txt:** Table of 7 bot names with status badges (allowed=green, blocked=red, not_mentioned=gray). Each row: bot name, status badge, rule line if available.
- **llms.txt:** Found/Not Found status, format validity indicator, validation errors list (if any), 500-char content preview in a code block.
- **Schema:** Detected types list with counts, raw format breakdown (JSON-LD / Microdata / RDFa counts), per-type details (type name, count, formats found).
- **Content:** Readability scores (Flesch + Fog raw values), text ratio (percentage), entity breakdown by type (ORG, PRODUCT, GPE, PERSON with counts), heading structure summary, Q&A density counts.

### Caching Contract (DASH-06)

Results persist across UI interactions via `st.session_state`:

- `st.session_state.pipeline_result` -- stores the dict returned by `run_pipeline()`
- `st.session_state.analysis_complete` -- bool flag, set True after first successful run
- Expander open/close, tab switches, and scroll do NOT re-trigger the pipeline
- Clicking "Analyze" again with the same URL re-runs; Streamlit's `st.button` fires on each click
- URL changes between runs are handled naturally: the new URL triggers a fresh pipeline run
